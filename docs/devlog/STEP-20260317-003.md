## Objective

Enforce fail-closed API auth/RBAC, require SUPER_ADMIN plus MFA for provider secrets, and migrate the local secret vault to PBKDF2-derived AES-256-GCM keys

## Change Scope

1. Rewrote API auth and RBAC defaults to fail closed:
   - `src/api/auth.js` now defaults `auth_enabled` to `true`, removes the anonymous `super_admin` fallback, requires a real bearer credential source, and returns `AUTH_LOCKDOWN` / `AUTH_MISCONFIGURED` instead of silently allowing requests.
   - `src/api/rbac.js` now defaults `rbac_enabled` to `true`, removes implicit role fallback when identity is missing, applies route-level default deny, and keeps `/integrations/im/commands` explicitly gated by role.
2. Hardened the task API ingress and secret settings surface:
   - `src/api/taskApiServer.js` now rejects all required-auth requests that are not successfully authenticated, returns `RBAC_LOCKDOWN` when RBAC is disabled or invalid, and requires `SUPER_ADMIN` plus MFA for `/settings/provider-secrets`.
   - `config/api_auth.json` and `config/rbac_policy.json` now initialize with auth/RBAC enabled.
3. Migrated the local vault to PBKDF2-derived AES-256-GCM keys:
   - `src/platform/secretVault.js` now derives keys with PBKDF2-SHA512 plus random salt, rotates with fresh salt, and upgrades legacy SHA-256-derived vault stores in place.
4. Completed related local-runner safety plumbing that the new tests depend on:
   - `src/platform/localExecutor.js`, `src/platform/policyStore.js`, and `src/orchestrator/taskStateMachine.js` now support case-insensitive path policy matching, `WAITING_FOR_AUTH` task suspension, `steps.jsonl` evidence logging, and non-workspace write authorization without breaking physical parent-directory checks.
5. Updated regression coverage:

## Risks

1. Enabling fail-closed auth/RBAC by default will lock existing local environments until a valid JWT secret or static token is configured.
2. Mitigation: the lockdown is explicit (`AUTH_LOCKDOWN`, `AUTH_MISCONFIGURED`, `RBAC_LOCKDOWN`) and the default config files now surface the secure posture instead of silently allowing anonymous access.
3. PBKDF2 vault migration rewrites legacy local vault files on first read.
4. Mitigation: migration is in-place, audited with `SECRET_VAULT_UPGRADED`, and regression coverage verifies legacy secrets remain readable after upgrade.
5. Provider secret access is now MFA-gated, which can block older admin scripts that only used bearer role claims.
6. Mitigation: static tokens and JWTs can carry `mfa_verified` / `mfa` claims, and the admin UI regression path now exercises the required shape.
